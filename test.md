# Открытые вопросы

## 1. Git

### 1.1. В общем случае, какие ветки чаще всего используются при работе над проектом?

*Ответ:*   
main - стабильный и протестированный код который работает на продакшене   
develop (dev) - основная ветка для разработки. После тестирования она отправляется в ветку main   

При работе с фичами или багами создаются отдельные временные ветки. После их Code Review через Pull Request (или Merge Request) они отправляются в develop.

---

### 1.2. Как отменить коммит?

*Ответ:*   
Если коммит локальный (не был отправлен на сервер)   
`git reset --soft HEAD~1` - коммит сотрется но файлы будут лежать в stage помеченные как измененные   
`git reset --hard HEAD~1` - полное удаление коммита и всех изменений в файлах   
`git commit --amend --no-edit` - последний коммит будет обновлен новыми изменениями без создания нового  
    
Если коммит был отправлен на сервер:   
`git revert хэш_коммита` - делает полностью противоположный коммит. История не переписывается.

## 2. JavaScript

### 2.1. Что такое Event Loop?

*Ответ:*   
это механизм, который позволяет однопоточному JavaScript выполнять асинхронные операции.   
Это бесконечный цикл, который проверяет стэк вызовов и очереди задач и выполняет их в определенном приоритете:
1. Стэк вызовов: текущий синхронный линейный код
2. Очередь микрозадач (промисы, MutationObserver, queueMicrotask)
3. Очередь макрозадач ( setTimeout, serInterval, события мыши и клавиатуры)
4. requestAnimationFrame 

Главное правило: Микрозадачи всегда важнее макрозадач. Они выполняются сразу, как только освободился стек.


---

### 2.2. Каким образом можно дождаться исполнения нескольких асинхронных операций, не блокируя тред/поток?

*Ответ:*   
Для этого есть стандартные методы для обработки массива промисов:   
- Promise.all() — Ждет выполнения всех промисов. Если хотябы в одном ошибка - возвращает ошибку.
- Promise.allSettled() — Всегда дожидается конца всех задач. Возвращает массив с результатами и статусами каждого промиса (успех/ошибка).   
- Promise.race() — Возвращает результат самого первого завершившегося промиса (успех или ошибку).   
- Promise.any() — Возвращает результат самого первого успешно выполнившегося промиса.

## 3. TypeScript

### 3.1. Чем отличается `any` от `unknown`?

*Ответ:*   
any - отключает проверку типов. Нужно избегать использования.
unknown - тип не известен. Заставляет сначала проверить тип (сузить). Как правило через условия проверки типов typeof или instanceof.

---

### 3.2. Что такое function overloading (перегрузка функций)?

*Ответ:*   
Это определение нескольких функций под одним именем, но с разными типами аргументов или их количества. Нативной поддержки этого нет в JS - только через ветвения и уточнения типов или количества аргументов.   
Для Typescript есть вариант реализации через сигнатуры для определения типов и отдельной реализации через ветвления.

```ts
// Сигнатуры
function TestFunc(id: number): string;
function TestFunc(name: string): string[];

// Сама функция
function TestFunc(param: number | string): string | string[] {
  if (typeof param === "number") return "Строка";
  return ["Массив", "Строк"];
}
```

## 4. Node.js

### 4.1. Поддерживает ли Node.js Event Loop?

*Ответ:*   
Да, поддерживает и использует. Главное отличие от браузер, что Node.js реализован на базе библиотеки libuv и разделен на другие специфические фазы:   
1. Таймеры (setTimeout и setInterval)
2. Отложенные калбеки (Ошибки сети)
3. Подготовка (cистемная фаза Node.js - не можем в ней что-то делать)
4. Опрос (Самая главная. Фаза ожидания, обработка событий, чтение файлов, сетевые запросы)
5. Проверка (setImmediate. спец фаза для выполнения кода после фазы опроса)
6. Закрывающие калбеки (закрытие соединений, например web сокетов )

Микрозадачи как и в браузере имеют наивысший приоритет и могут выполняться между любыми фазами. К ним относятся промисы и process.nextTick (наивысший приоритет, выполняется раньше промисов).


---

### 4.2. При работе со стандартной библиотекой Node.js, как использовать современные функции, работающие на Promise, а не callback'ах?

*Ответ:*   
Использовать встроенный модуль /promises (например node:fs/promises). Основные модули fs, dns, crypto, timers.

## 5. React.js

### 5.1. Как совершить единовременный HTTP-запрос при рендеринге компонента?

*Ответ:*   
Для этого нужно вызывать хук useEffect с пустым массивом зависимостей [].

```ts
useEffect(() => {  
    // Обрабатываем случай для Race Conditions при закрытии компонента до получения ответа
    const controller = new AbortController();

    const fetchData = async () => {
      try {
        setLoading(true);
        const response = await fetch('https://example.com', { signal: controller.signal });
        if (!response.ok) throw new Error('Ошибка загрузки данных');
        
        const data: Data = await response.json();
        setUser(data);
      } catch (err) {
        setError(err instanceof Error ? err.message : 'Ошибка');
      } finally {
        setLoading(false);
      }
    }; 
    fetchData();
    // Функция очистки запроса будет вызывана при размонитровании компонента 
    return () => controller.abort();
  }, []);

```

### 5.2. Что такое key (ключ), и зачем он нужен?

*Ответ:*   
Помогает React идентифицировать элементы: понимать, какие из них изменились, были добавлены или удалены при обновлении списка. Для оптимизации ключ должен быть стабильным между рендерами. Использовать индексы массива - плохо. Нужна уникальная строка в идеале id из БД. Если изменяем ключ - то компонент пересоздается (Помогает вручную пересоздать компонент при необходимости)

## 6. Postgres

### 6.1. Чем отличаются `LEFT JOIN` и `INNER JOIN`?

*Ответ:*   
INNER JOIN - Возвращает только те строки, для которых есть совпадения в обеих таблицах.   
LEFT JOIN - Возвращает абсолютно все строки из левой таблицы, а из правой таблицы добавляет только те, для которых нашлось совпадение   
---

### 6.2. Что такое индекс? Зачем нужно индексировать колонки/поля в таблицах?

*Ответ:*   
Индексы помогают избежать полного сканирования таблицы при поиске (Seq Scan). Ускоряют ORDER BY, GROUP BY  и JOIN.
Но индексы замедляют операции INSERT, UPDATE, DELETE и увеличивают расход дискового пространства.
Важно индексировать только те поля по которым идут часто операции WHERE, JOIN, ORDER BY или GROUP BY и таблица имеет много записей. 
Для Postresql можно индексировать и JSONB поля, для индекса по умолчанию используется бинарное дерево. Так есть вариант использования составных индексов, где важен порядок колонок.

- генерация множества файлов
- нарушение структуры проекта 
- "галлюцинации" и циклическая генерация не рабочего кода

# Тестовое задание

Создайте **offline-first PWA трекер расходов** на TypeScript, используя React и Node.js.

Исходный код должен быть Git-репозиторием. Варианты сдачи:

1. Публичный хостинг репозиториев (GitHub / GitLab / Codeberg / etc.)
2. Сжатый архив (предпочтительно `tar.gz`)

---

## 1. Backend

Бэкенд на Node.js + Express.js.

### 1.1. База данных

- Подключение к PostgreSQL
- Конфигурация через environment variables

### 1.2. Таблица статей расходов

Используя любой коннектор/ORM, создать таблицу со следующими полями:

| Поле   | Описание                  |
|--------|---------------------------|
| `id`   | Уникальный идентификатор  |
| `name` | Название статьи           |
| `sum`  | Сумма статьи              |
| `date` | Дата статьи               |

> Тип данных, дополнительные поля и прочее - на усмотрение разработчика.

---

## 2. Frontend

React + Vite.

### 2.1. Хедер

Статус подключения к сети / бэкенду с живым апдейтом.

### 2.2. Форма создания статьи расходов

| Поле            | Валидация                        |
|-----------------|----------------------------------|
| Название статьи | 3 ≤ x ≤ 30 символов              |
| Сумма           | 1.00 ≤ x ≤ 1 000 000.00 у.е.    |
| Дата            | Любая                            |

### 2.3. Список статей

Рядом с формой - список созданных статей с итоговой суммой расходов. Бонусом будет:
- Корректный расчет суммы у.е.
- Имплементация удаления уже созданных статей
- Функционал редактирования данных уже созданных статей делать не обязательно

### 2.4. Offline-first

При отсутствии подключения созданные статьи сохраняются локально. После восстановления соединения - автоматическая синхронизация с бэкендом.

> Стиль, библиотеки компонентов и прочее - на усмотрение разработчика.
